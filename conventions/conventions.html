Naming

void MethodName() { }

Methods I find, in some way, immutable and not like a variable you can change to give some other result to the rest of the exucution.

Class Foo { };

Not declaring classes with uppercase first letter makes your code start to get wierd. Is it a type, class, dimmer. Write classes with uppercase.

Foo* foo = new Foo;

However the instantiation will have to be all lower case because we now have a type we can do some changing on. Every instance of classes, types and so on, just keep it lower case.

const int MAX_DIMMER = 1337;

All constants must all be uppercase and using underscore to seperate words. However try minimizing the use of constants. Some times implementing a getter method is better:

int GetMaxDimmer() { return 1337; } // Also takes up the same amount of space.

char* textstringforprinting;

Instances of different types, should be in lowercase, those of you using different IDE tool don't mind camelcase(textStringForPrinting), but the rest of us has to live with always has to misclicking the shift button for caps or even control if you're about to enter the 4am point at night.

{Braces}

First of all, I've been told there is a lot of confusion how to name [] brackets and braces {}, some countries switch these, even call them the same. Therefore to avoid any confusion, from here on, brackets are squared [] and braces are curled {}.

There is alot of arguing where the braces should go, I'll show both and I don't mind using either, but there's pros and cons for both.

int main(void) {
    int foo = 0;
    return foo;
}

First of all this saves you one line of code, which makes your code look more compact. Some might like the more traditional way of putting the starting brackets in its own line. However some like to comment out different loops to check a simple run of the code inside therefore this might not work as wonderess.

int main(void)
{
    int foo = 0;
    return foo;
}

The good thing about this style is that the braces are aligned, it's very easy to see where the method main is ended. For some that might be too intuitive and will not see the advantage of doing it like this.

But for those of you who like your code to be symmetric and pretty. Insert a bunny and go code, I don't mind this at all. //But the bunny will have to be there


When you have a statement, i.e. using an if statement. Should you be using braces or no braces. I prefer to use the two styles written below.

if (condition) { statement; }

This has some advantages. It's only one line of code and it's clean to read. However if your statement or condition takes up too much linespace, I prefer using this second choice.

if (condition)
    statement;

However never use braces in this one line statement. It just takes up an ekstra line of code, and may lower the readability of your code.


(Parenthesis)

Method calls have their parenthesis close to the name, as I don't think it makes any sense to have them anywhere else. MethodName(param1);

When writing ifs you should keep white space before and after the parenthesis to let it be easy to spot the start and the end of your if condition.

if ( GetNumber(index--) <= this->id )
